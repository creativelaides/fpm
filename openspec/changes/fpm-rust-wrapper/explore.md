# Exploration: fpm-rust-wrapper

## Executive Summary

`fpm` will be an open-source Rust CLI for Windows that mirrors the ergonomic shape of `fnm` (Fast Node Manager) but for Python. It wraps the official Python Install Manager (`py` / `pymanager`) and replaces the PowerShell `$profile` block the user currently uses to switch Python versions. The first slice targets Windows only, using `py list`/`py -V:<tag>` under the hood and emitting shell integration through `fpm env --shell powershell`.

---

## Current State

- The project is **greenfield** — no source code, `Cargo.toml`, or test runner exists yet.
- The user's PowerShell `$profile` already contains a hand-written Python version manager implemented in PowerShell. It intercepts subcommands like `py use`, `py default`, `py current`, `py list`, `py install`, etc., delegating everything else to `py.exe`. This is the exact surface `fpm` must replace.
- The active Windows environment has the official **Python Install Manager** installed (`py.exe` at `%LocalAppData%\Microsoft\WindowsApps\py.exe`, version 26.3, Python 3.14.6 registered as the default runtime).
- `cargo`/`rustc` 1.96 are available, so the first implementation task can scaffold a Rust project immediately.

---

## Affected Areas

- `src/main.rs`, `src/cli.rs` — entry point and clap subcommand routing (`use`, `list`, `current`, `env`, `default`, `install`, `uninstall`, `exec`, `completions`).
- `src/commands/*.rs` — command implementations. `use`, `list`, `current`, `env` are the first-slice commands.
- `src/shell/powershell.rs` — PowerShell `env` integration script generator (PATH shim + `use-on-cd` hook).
- `src/pymanager.rs` (new module) — abstraction over `py list --format=json`, `py -V:<tag>`, and `pymanager.json` config reads/writes.
- `src/config.rs` — `FPM_DIR`, log level, default tag strategy, Windows-specific overrides.
- The user's `%UserProfile%\Documents\PowerShell\Microsoft.PowerShell_profile.ps1` — must replace the existing PowerShell `py { ... }` wrapper block with `fpm env --use-on-cd --shell powershell | Out-String | Invoke-Expression`.

## fnm Reference Architecture

### Command Surface

fnm exposes the following commands (from `src/cli.rs` / `docs/commands.md`):

| Command | Alias | Purpose |
|---|---|---|
| `list-remote` | `ls-remote` | List remote Node.js versions |
| `list` | `ls` | List locally installed versions |
| `install` | `i` | Install a new version |
| `use` | — | Change active version |
| `env` | — | Print shell setup commands |
| `completions` | — | Print shell completions |
| `alias` / `unalias` | — | Manage aliases |
| `default` | — | Set / get default version |
| `current` | — | Print current version |
| `exec` | — | Run a command within fnm context |
| `uninstall` | `uni` | Uninstall a version |

### Version-Switching Mechanism

fnm does **not** mutate global PATH. It creates a **per-session symlink directory** (`multishell_path`) and prepends it to PATH via the shell integration generated by `fnm env`:

1. `fnm env` creates a unique symlink directory pointing to the selected version's installation dir.
2. It prints shell commands that:
   - Set `FNM_MULTISHELL_PATH` to that directory.
   - Prepend it to `PATH` (on Windows the directory itself; on Unix `.../bin`).
   - Export other config env vars (`FNM_DIR`, `FNM_LOGLEVEL`, etc.).
3. `fnm use <version>` resolves the version, then **re-targets the symlink** (`replace_symlink` in `src/commands/use.rs`) so the same `PATH` entry now points to a different version.

Key implementation details from `src/commands/env.rs` and `src/commands/use.rs`:

- Uses `std::env::set_var` only for the internal `use-on-cd` initial call.
- Relies on the parent shell to evaluate the emitted script for normal usage.
- Aliases are also symlinks under `$FNM_DIR/aliases/<name>`.

### Shell-Integration Model

- `fnm env --shell <shell>` emits shell-specific text. Supported shells: `bash`, `zsh`, `fish`, `powershell`, `cmd` (Windows).
- PowerShell integration (`src/shell/powershell.rs`) prints:
  - `$env:PATH = "<multishell_path>;$env:PATH"`
  - `$env:FNM_* = ...`
  - With `--use-on-cd`: a hook that overrides `Set-Location` and calls `fnm use --silent-if-unchanged` when a `.python-version` (analogous to `.node-version`) exists.
- The README tells users to add: `fnm env --use-on-cd --shell powershell | Out-String | Invoke-Expression`

### Config File Approach

- Config is passed primarily through **environment variables** and **global CLI flags** (all flattened into the top-level `FnmConfig`).
- Persistent installation root is `$FNM_DIR` (defaults to `%LocalAppData%/fnm` via `etcetera` + `directories.rs`).
- Version files: `.node-version`, `.nvmrc` (and optionally `package.json#engines.node`).

## PyManager / `py` Surface on Windows

### What `py` Actually Is

The modern Windows Python Install Manager (PyManager) ships the commands `python`, `py`, and `pymanager`. `py` is the multi-version-aware launcher. `pymanager` is the unambiguous management command (`py` without args launches the default interpreter; `pymanager` without args shows help).

Key facts from `docs.python.org/3/using/windows.html` and local verification:

- `py -V:<TAG>` launches a specific runtime. For official CPython the tag looks like `3.14`, `3.14-64`, `3-arm64`, etc. The `V:` can be omitted for tags starting with `3`.
- `py -3.14` is shorthand for `py -V:PythonCore\3.14`.
- `py list` shows installed runtimes. Formats: `table`, `csv`, `json`, `jsonl`, `exe`, `prefix`.
- `py list --one <TAG>` returns the single runtime that `py -V:<TAG>` would launch.
- `py install <TAG>` downloads and installs a runtime.
- `py uninstall <TAG>` removes it.
- `py exec <args>` installs if missing (when `automatic_install` is true).

### What fpm Must Shell Out To

| fpm command | Recommended underlying call |
|---|---|
| `fpm list` | `py list --format=json` (or `jsonl` for streaming) |
| `fpm current` | `py -V` or `py -c "import sys; print(sys.version)"` |
| `fpm use 3.14` | Set `PYTHON_MANAGER_DEFAULT=3.14` for the session; optionally persist `default_tag` in `pymanager.json` |
| `fpm default 3.14` | Write `"default_tag": "3.14"` into `%AppData%\Python\pymanager.json` |
| `fpm exec --using=3.14 python ...` | `py -V:3.14 ...` or `pymanager exec -V:3.14 ...` |
| `fpm install 3.14` | `py install 3.14` |
| `fpm uninstall 3.14` | `py uninstall 3.14` |

### What fpm Can Read Directly

- **Registry (PEP 514)**: `HKEY_CURRENT_USER\Software\Python\<Company>\<Tag>\InstallPath` gives the install path and executable. PyManager also writes these via `pep514` shortcuts.
- **Config file**: `%AppData%\Python\pymanager.json` (JSON) stores `default_tag`, `default_platform`, `automatic_install`, etc.
- **Install root**: `%LocalAppData%\Python\<company>-<tag>` is the observed layout (e.g., `C:\Users\devel\AppData\Local\Python\pythoncore-3.14-64`).

### Important Gotchas

1. **Per-user vs all-users installs**: PyManager installs runtimes **per-user only**. The registry root is `HKEY_CURRENT_USER\Software\Python` by default. There is no per-machine runtime install in PyManager.
2. **PATH ordering**: The actual `python`/`py` commands are Windows app execution aliases in `%LocalAppData%\Microsoft\WindowsApps`. They must remain on PATH. `fpm` should not prepend a generic `python.exe` shim that could shadow them.
3. **`py.ini` is deprecated**: PyManager ignores the legacy `py.ini` file. Configuration is JSON via `pymanager.json` or env vars (`PYTHON_MANAGER_DEFAULT`, `PYTHON_MANAGER_DEFAULT_PLATFORM`, etc.).
4. **Version selection precedence**: Active virtual environment > shebang > `PYTHON_MANAGER_DEFAULT`/`default_tag` > latest installed.
5. **Performance of `py` calls**: Local measurements show `py -3.14 -c ...` takes ~190 ms, `py list --format=json` takes ~150 ms. A Rust wrapper can be faster than the PowerShell wrapper by avoiding PowerShell function overhead, but it still incurs `py` spawn cost. Caching the `py list` JSON for the lifetime of a command is essential.

## PowerShell `$profile` Replacement Contract

### What the Current `$profile` Block Does

The existing block (in `%UserProfile%\Documents\PowerShell\Microsoft.PowerShell_profile.ps1`):

- Defines a function `py` that intercepts `use`, `default`, `current`, `install`, `uninstall`, `list`, `exec`, and passes everything else to `py.exe`.
- For `py use <tag>`: validates the tag against `pymanager list`, sets `$env:PYTHON_MANAGER_DEFAULT`, and prints a confirmation.
- For `py default <tag>`: writes `default_tag` into `$env:AppData\Python\pymanager.json`.
- For `py current`: runs `py.exe -V`.
- Does **not** modify PATH directly; relies on `py.exe` to resolve versions via `PYTHON_MANAGER_DEFAULT`.

### What `fpm` Must Satisfy

`fpm` must be a drop-in replacement for the PowerShell `py { ... }` wrapper:

- `fpm use 3.14` should switch the Python version for the current PowerShell session.
- `fpm default 3.14` should persist the default version.
- `fpm current` should show the active version.
- `fpm list` should show installed runtimes.
- Any unrecognized `fpm <args>` should be passed through to `py.exe` so existing aliases like `mkdx { py -3.13 -m markitdown }` keep working if the user later rewrites them to `fpm -3.13 -m markitdown`.

### Recommended Shell-Integration Approach

Adopt fnm's model but simplified for Windows/PowerShell:

1. Add to `$profile`:
   ```powershell
   fpm env --use-on-cd --shell powershell | Out-String | Invoke-Expression
   ```
2. `fpm env` emits:
   - `$env:FPM_DIR = ...`
   - `$env:PATH = "<fpm-shim-dir>;$env:PATH"` where `<fpm-shim-dir>` is a per-session directory containing a `python.exe` shim.
   - Optionally a `Set-Location` hook for `--use-on-cd` that calls `fpm use --silent-if-unchanged` when `.python-version` exists.
3. `fpm use 3.14` resolves the real `python.exe` path via `py list --one --format=exe 3.14` and rewrites the shim in `<fpm-shim-dir>` to point to it.

### Alternative: No Shim, Just Mutate `PYTHON_MANAGER_DEFAULT`

| Approach | Pros | Cons | Effort |
|---|---|---|---|
| **A. Per-session PATH shim (recommended)** | Mirrors fnm exactly; `python` resolves through fpm; works with tools that call `python` directly | Needs a writable shim directory per session; must handle `pythonw.exe` too | Medium |
| **B. Mutate `PYTHON_MANAGER_DEFAULT` env var** | Very simple; matches current PowerShell profile behavior | Only affects `py`/`python` resolution, not direct `python.exe` calls from tools; no PATH change | Low |
| **C. Mutate `pymanager.json` on every `use`** | Persists across sessions | Writes disk on every shell command; slower; races between shells | Medium |

**Recommendation**: Use **Approach A** for the design, but implement `use` first via **Approach B** as the MVP because it is the minimal safe replacement for the current `$profile` block. The shim can be added in a follow-up slice without changing the CLI contract.

## Rust Stack Recommendation

Given the greenfield state, the first `cargo init` should include these crates:

| Crate | Purpose | Rationale |
|---|---|---|
| `clap` v4 with `derive` + `env` + `cargo` | CLI parsing, subcommands, global flags, completions | Matches fnm's `cli.rs` structure; gives `--help` and shell completions for free. |
| `serde` + `serde_json` | Deserialize `py list --format=json` output and read/write `pymanager.json` | PyManager speaks JSON; PEP 514 registry values are strings. |
| `winreg` | Read Windows registry (PEP 514 `HKEY_CURRENT_USER\Software\Python`) | Lightweight, pure-Rust enough for registry enumeration. |
| `tokio` (or `std::process`) | Async not strictly needed; `std::process::Command` for spawning `py` | Keep it synchronous for v1 to avoid complexity; spawn `py list`/`py -V`. |
| `anyhow` / `thiserror` | Error handling | `thiserror` for domain errors; `anyhow` for command glue. |
| `etcetera` | Cross-platform dirs (used here mainly for Windows data dir fallback) | fnm uses it; on Windows defaults to `%LocalAppData%`. |
| `colored` (optional) | Colored terminal output | Matches fnm's UX; can be deferred. |
| `tempfile` | Create per-session shim directories safely | Already used by fnm for multishell dirs. |
| `junction` or `symlink` | Windows directory junctions for shims | Only if implementing shim approach; `std::os::windows::fs::symlink_file` works for file shims. |
| `which` (optional) | Verify `py.exe` / `pymanager.exe` on PATH | Small helper; can also use `Get-Command` equivalent in Rust via env PATH scan. |


### Windows-Specific Notes

- Build target: `x86_64-pc-windows-msvc` (the host here is AMD64 Windows).
- Registry access: use `winreg` with `HKEY_CURRENT_USER` first, then `HKEY_LOCAL_MACHINE` + `KEY_WOW64_32KEY` for legacy compatibility per PEP 514.
- Process spawning: prefer `std::process::Command::new("py")` over hard-coded paths; `py.exe` is on PATH via `%LocalAppData%\Microsoft\WindowsApps`.

## Scope: IN vs OUT (First Slice)

### IN Scope

- Windows only.
- Commands:
  - `fpm use <version>` — switch active Python version for the current shell session.
  - `fpm list` — list installed runtimes (wraps `py list --format=json`).
  - `fpm current` — show active Python version.
  - `fpm env --shell powershell` — emit PowerShell setup script.
  - `fpm default [version]` — get/set persistent default version via `pymanager.json`.
  - `fpm --version` / `--help`.
- PowerShell `$profile` integration: replace the existing hand-written `py { ... }` block with `fpm env --use-on-cd --shell powershell | Out-String | Invoke-Expression`.
- Calling `py` / `pymanager` under the hood; no direct Python downloads in v1.

### OUT of Scope

- macOS / Linux (the user explicitly said "solo windows").
- Other shells (cmd, bash, zsh, fish) for v1.
- Installing Python from scratch (`fpm install` may be a thin wrapper in a later slice, but not the primary deliverable now).
- Virtual environment management.
- Multi-arch / 32-bit detection and switching.
- `fpm completions`, `fpm alias`, `fpm exec` in the first slice; keep them on the roadmap.
- Direct registry-based version switching without invoking `py`; we delegate to `py` to avoid reimplementing PyManager's resolution logic.

## Risks

1. **PATH shadowing**: If `fpm` creates a `python.exe` shim and places it before `%WindowsApps%`, it could break Windows app execution aliases. Mitigation: only put the shim directory at the front of PATH inside the `fpm env` generated script, and ensure the shim forwards to the real `py.exe`-resolved executable.
2. **Per-user vs all-users installs**: PyManager is per-user, but the user may have legacy all-users python.org installs. `py list --format=json` already merges managed and unmanaged runtimes by default; fpm should rely on that rather than reimplementing registry merge logic.
3. **PyManager version differences**: The legacy `py.exe` launcher and modern PyManager behave differently. `fpm` should target PyManager 26.x+ (the current installed version) and document that older launchers are unsupported.
4. **UAC / writable config**: Writing to `%AppData%\Python\pymanager.json` does not require admin. There is no per-machine install support in PyManager, so UAC is not in scope for v1.
5. **Performance ceiling**: Each `py list` call costs ~150 ms. The Rust binary itself is fast, but the underlying PyManager calls dominate. We must cache the parsed runtime list for the lifetime of each `fpm` invocation and avoid multiple `py` spawns.
6. **Session-only vs persistent `use`**: The current `$profile` `py use` only sets `$env:PYTHON_MANAGER_DEFAULT`. If the user expects `fpm use` to persist like `fpm default`, that is a UX ambiguity to resolve.
7. **PowerShell profile maintenance**: Replacing the existing block means `fpm` must be installed and on PATH before the profile change, or the new profile line will fail on shell startup.

## Open Questions (Need User Clarification Before Proposal)

1. **Install vs wrap**: Should `fpm install <version>` actually download Python itself, or only call `py install <version>`? The current PowerShell wrapper does the latter.
2. **Persistence of `use`**: Should `fpm use 3.14` persist as the default (write `pymanager.json`), or only affect the current session? The current `$profile` `py use` is session-only; `py default` is persistent.
3. **Pass-through semantics**: Should unrecognized `fpm <args>` be forwarded to `py.exe` exactly like the current wrapper does, or should `fpm` be a strict subcommand CLI and the user explicitly calls `py` for raw Python? This affects whether aliases like `mkdx` need rewriting.
4. **Shim vs env-var switching**: Do you want the eventual `fpm env` + shim approach (so `python` resolves through fpm), or is mutating `PYTHON_MANAGER_DEFAULT` sufficient for v1?
5. **Profile editing**: Can the apply phase directly edit your `$profile`, or must it generate a manual snippet for you to paste?
6. **Default version file**: Should `fpm use` (with no argument) read a `.python-version` / `pyproject.toml` requirement, analogous to fnm's `.node-version`? If yes, which files?

## Recommendation

Proceed to **sdd-propose** with the following shape:

- Build a Rust CLI named `fpm` using `clap`, `serde_json`, and `winreg`.
- First slice implements `use`, `list`, `current`, `default`, `env --shell powershell`, and `--version`.
- `fpm use <version>` sets `PYTHON_MANAGER_DEFAULT` for the current session; `fpm default <version>` persists it to `pymanager.json`.
- `fpm list` parses `py list --format=json` and renders a friendly table.
- `fpm env` emits a PowerShell script that adds an `fpm`-managed shim directory to PATH and optionally hooks `Set-Location` for `--use-on-cd`.
- The apply phase scaffolds `Cargo.toml`/`src/` and updates the user's `$profile` to replace the existing Python wrapper block.

This is the lowest-risk, highest-fidelity replacement for the existing PowerShell wrapper and directly mirrors fnm's ergonomics while delegating all Python runtime management to the official PyManager.
